# WARDEN Implementation Guide

**Product Name:** WARDEN (Wireless Advanced Response & Deployment Network)
**Document Version:** 1.0
**Date:** June 2026

---

## 1. Introduction
This Implementation Guide details how the WARDEN ecosystem is constructed from the ground up. It defines the core modules (sub-projects), their exact responsibilities, how they connect with each other, and the sequential approach required to build and deploy the complete platform.

## 2. Core Sub-Projects & Responsibilities

WARDEN is a distributed system consisting of five primary sub-projects. Implementing the full system requires developing these nodes in parallel or sequentially as defined below.

### 2.1 The Central Hub: Backend (FastAPI)
*   **Role:** The central router and brain of the operation.
*   **Implementation Requirements:**
    *   **Environment:** Python 3.11+, Uvicorn, FastAPI.
    *   **Core Modules:** 
        *   `WebSocket Manager`: Handles persistent, low-latency connections with all clients for real-time alert propagation.
        *   `Auth Middleware`: Validates JWTs (`HS256`) for every incoming REST and WebSocket request.
        *   `AI Router`: An asynchronous module that communicates with Google Gemini API for high-level tactical summaries, and local Vector Databases (via `vectorDB_service.py`) for RAG context retrieval.
        *   `Database Controller`: Wraps the Firebase Admin SDK to write/read persistent operational data (Incidents, Tasks).

### 2.2 The Environment: Simulation Engine (Node.js)
*   **Role:** Simulates the physical environment and occupant telemetry.
*   **Implementation Requirements:**
    *   **Environment:** Node 18/20, TypeScript, managed via a Turborepo workspace.
    *   **Core Modules:**
        *   `Physics Engine`: Calculates hazard spread (e.g., heat dispersion, smoke propagation) based on predefined building geometry.
        *   `Occupant Tracker`: Simulates occupant movement through BLE/Wi-Fi node check-ins.
        *   `Pathfinding Service`: Calculates safe routes (A* or Dijkstra variations) that dynamically avoid hazard zones generated by the Physics Engine.

### 2.3 Command Center: Staff Desktop App (Rust/Tauri)
*   **Role:** High-level dashboard for Command Staff.
*   **Implementation Requirements:**
    *   **Environment:** Rust (backend logic, OS integration) and React/TypeScript (frontend).
    *   **Core Modules:**
        *   `Map Renderer`: A highly performant 2D/3D canvas (e.g., WebGL/Three.js) to render the digital twin.
        *   `Local AI Sidecar`: Integration with a local LLM runner (like `llama.cpp` or ONNX runtime) executing Gemma-2b for offline tactical support.
        *   `ICS Task Board`: A real-time synchronized Kanban board for unit dispatch.

### 2.4 Tactical Node: Responder Desktop App (Rust/Tauri)
*   **Role:** Rugged, field-ready dashboard for first responders.
*   **Implementation Requirements:**
    *   **Environment:** Rust and React.
    *   **Core Modules:**
        *   Shares the `Local AI Sidecar` module with the Staff App for complete offline resilience in concrete/steel environments without network connectivity.
        *   `Task Receiver`: A specialized, simplified UI that only displays active assignments and immediate local hazards, preventing cognitive overload.

### 2.5 Mobile Endpoint: Warden Android App (Kotlin)
*   **Role:** Civilian and responder mobile interface.
*   **Implementation Requirements:**
    *   **Environment:** Android SDK 34+, Kotlin, Jetpack Compose.
    *   **Core Modules:**
        *   `SOS Broadcaster`: A high-priority service that bypasses standard queues to blast GPS/Indoor location data to the FastAPI backend.
        *   `Indoor Navigator`: Renders the dynamic safe-paths pushed by the Backend.
        *   `Secure Chat`: Real-time chat interface connected directly to the Command Staff.

### 2.6 Public Website (Vite / React / Tailwind)
*   **Role:** Public informational portal and landing page for the WARDEN ecosystem.
*   **Implementation Requirements:**
    *   **Environment:** Node.js, React 19, Vite, Tailwind CSS 4.
    *   **Core Modules:**
        *   `Landing Page`: High-level overview of the WARDEN system capabilities.
        *   `App Distribution`: Download links for the Android APK and Desktop installer binaries.

## 3. System Connections & Data Pipelines

The true power of WARDEN lies in how these sub-projects are connected:

1.  **Simulation -> Backend (Data Ingestion):** 
    *   The Node.js Simulation Engine pushes environmental updates to the FastAPI backend via a dedicated internal REST endpoint or binary WebSocket stream.
2.  **Backend <-> Clients (Bi-directional Sync):** 
    *   All UI clients (Staff, Responder, Mobile) maintain a persistent WebSocket connection to the Backend. When the backend receives a hazard update, it broadcasts it to the clients in <50ms.
3.  **Clients -> Backend -> AI (Intelligence Loop):** 
    *   A Staff user requests a tactical summary -> Backend searches VectorDB for context -> Backend queries Gemini API -> Backend streams the markdown response back to the Staff UI over WebSockets.
4.  **Clients -> Local AI (Offline Loop):** 
    *   If network connection is severed, Desktop Apps route queries locally to the embedded Gemma model via Tauri's Rust-to-JS bridge, bypassing the FastAPI backend entirely.

## 4. Implementation Phases

To build WARDEN from scratch, development should follow these phases:

### Phase 1: Infrastructure & Data Models
*   Set up Firebase/Firestore schemas (Users, Incidents, Tasks).
*   Create the Python backend skeleton.
*   Implement JWT authentication and the foundational WebSocket manager.

### Phase 2: Simulation & Rendering
*   Build the Node.js Turborepo.
*   Develop the basic Physics Engine (simulating a fire on a grid).
*   Build the Staff Desktop App map canvas to listen to backend WebSockets and render the grid.

### Phase 3: AI & Routing
*   Integrate Gemini API and the Vector Database in the backend.
*   Implement the Pathfinding Service in the Node.js engine to route around simulated hazards.
*   Embed the local Gemma-2b models into the Tauri applications.

### Phase 4: Edge Clients & Field Ops
*   Develop the Responder Desktop App UI (consuming ICS tasks).
*   Build the Android App, focusing on the SOS trigger and push-notification routing.

### Phase 5: Hardening
*   Optimize WebSocket payload compression for low-bandwidth scenarios.
*   Ensure the local AI sidecars failover seamlessly when cloud connection drops.
