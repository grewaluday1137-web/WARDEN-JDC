# WARDEN Data Schema & Information Flow

**Product Name:** WARDEN (Wireless Advanced Response & Deployment Network)
**Document Version:** 1.0
**Date:** June 2026

---

## 1. Introduction
This document outlines the data storage strategies, database schemas, and the lifecycle/flow of data payloads throughout the WARDEN ecosystem. 

## 2. Data Storage Strategy
Data within WARDEN is categorized into two primary forms:
1.  **Persistent Operational Data:** Stored in **Firebase / Firestore** (NoSQL). This includes user profiles, incident logs, active command tasks, and historical metadata.
2.  **Vector / Contextual Data:** Stored in the backend's **Vector Database** (`vectorDB_service.py`). This includes high-dimensional embeddings representing floor plans, safety protocols, and past incident reports to power the AI reasoning relay.
3.  **Transient Telemetry:** High-throughput data like live occupant coordinates or simulated heat spread is routed entirely in-memory via the FastAPI WebSocket hub, only logging aggregated snapshots to the database to prevent write-fatigue.

## 3. Core Database Schemas (Firestore / NoSQL)

### 3.1 Users Collection
Stores profiles for command staff, field responders, and registered building occupants.
```json
{
  "uid": "string (Firebase Auth UID)",
  "role": "string (admin | responder | occupant)",
  "name": "string",
  "contact": "string",
  "current_location": {
    "floor": "number",
    "lat": "number",
    "lng": "number",
    "last_updated": "timestamp"
  },
  "status": "string (safe | in_danger | unknown)"
}
```

### 3.2 Incidents Collection
The core record of an active or historical emergency.
```json
{
  "incident_id": "string",
  "type": "string (fire | medical | security)",
  "severity": "string (low | medium | critical)",
  "epicenter": {
    "floor": "number",
    "sector": "string"
  },
  "status": "string (active | contained | resolved)",
  "ai_summary": "string (Generated by Google Gemini)",
  "created_at": "timestamp",
  "resolved_at": "timestamp (nullable)"
}
```

### 3.3 ICS (Incident Command System) Tasks Collection
Represents actionable assignments generated by Command Staff or AI for the Responders.
```json
{
  "task_id": "string",
  "incident_id": "string (ref -> Incidents)",
  "assigned_to": "string (ref -> Users.uid)",
  "objective": "string (e.g., 'Evacuate Sector B')",
  "priority": "number (1-5)",
  "status": "string (pending | in_progress | completed)",
  "created_at": "timestamp"
}
```

## 4. Vector Database Schema
The Vector DB holds documents embedded for similarity search (RAG) by the Gemini/Gemma AI sidecars.
*   **Safety Protocols:** Text chunks of standard operating procedures.
*   **Floor Plan Metadata:** Embedded structural notes (e.g., "Sector C contains hazardous materials").
*   **Historical Incidents:** Embedded summaries of past resolutions for predictive AI suggestions.

---

## 5. End-to-End Data Flow

### Step 1: Ingestion & Trigger
*   **Source:** A Mobile Client sends an SOS, or the Simulation Engine triggers a hazard alert.
*   **Transport:** The payload is sent via Secure WebSocket or HTTP POST to the FastAPI Backend.
*   **Action:** The Backend immediately parses the raw JSON payload and authenticates the origin using JWT.

### Step 2: Processing & Enrichment
*   **State Update:** The Backend writes the new Incident state to Firestore (creating a new document in the `Incidents` collection).
*   **AI Context Retrieval:** The Backend queries the Vector DB using the hazard location to fetch spatial risks.
*   **AI Inference:** The Backend sends the incident payload + Vector context to the Google Gemini API to generate an `ai_summary` and tactical recommendations.

### Step 3: Distribution & Synchronization
*   **Broadcast:** The Backend blasts a highly compressed WebSocket payload containing the updated `Incidents` data and the new AI-generated safe routes to all subscribed clients (Staff App, Responder App, Mobile App).
*   **Client Rendering:** The Desktop Tauri apps immediately render the new hazard zones on the 2D/3D map canvas, while the Mobile App transitions to the active evacuation UI.

### Step 4: Resolution & Archival
*   **Task Updates:** Responders mark their ICS Tasks as `completed`. This updates the Firestore document, which triggers a real-time sync back to the Staff Desktop App.
*   **Closure:** Once the incident is marked `resolved`, the Backend packages the timeline, embeds it via an embedding model, and stores the resulting vectors in the Vector DB to improve future AI predictions.
